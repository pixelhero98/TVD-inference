# TVD-Scheduler Lessons

- Isambard3 MACS Slurm jobs defaulted to `ReqMem=1G`, which OOM-killed even a tiny OTFlow forecast smoke run during dataset/model setup. Set explicit memory on evaluation jobs; `--mem=128G` fixed the baseline smoke.
- Importing `torch` from the project venv can hang on the Isambard3 login node, while the same import succeeds quickly inside a `hopper` Slurm job with GPU allocation. Run environment validation through Slurm preflight instead of login-node Python imports.
- Running `python -m unittest code/test_*.py` from the repo root collides with Python's stdlib `code` module. Run targeted tests from `code/` with module names, e.g. `cd code && ../.venv/bin/python -m unittest test_otflow test_otflow_paper_prep`.
- The canonical study rewrites `rows.jsonl` and `rows.csv` from an in-memory row table on each append. Stop the live runner before filtering invalid rows, then restart from the cleaned files so stale rows are not reintroduced.
- Resume checks must happen before canonical calibration/evaluation. Build the expected row signatures for each `dataset x seed x target_nfe x solver` group first; if all scheduler cases are already complete, skip calibration entirely.
- Recovery runs that reuse frozen `uniform` evidence should read an external comparator CSV and compute relative gains from same-key `seed x dataset x solver x NFE x checkpoint` matches, rather than regenerating baselines in a new output root.
- `_pending_scheduler_cases(...)` should default raw TVD cases to the canonical signal key when `signal_trace_key` is omitted; tests and resume logic otherwise disagree on the expected row signature.
- LOB rollout diagnostics should align context features from runtime tensor shapes, not only `cfg.snapshot_dim`; checkpoint metadata can lag the sample block width and cause `torch.cat` feature mismatches after a completed evaluation.
- Published fixed schedules must be mapped through their native coordinate before normalization. GITS uses sigma nodes directly, AYS uses DDPM timestep indices, and OTS uses optimized VP times; log-sigma remapping makes the baseline table unfair.
- The local Windows Python may not have `torch`; use `ssh yzn3090 "cd ~/work/TVD-Scheduler/code && ../.venv/bin/python -m unittest ..."` for scheduler tests that import OTFlow modules.
- TVD mass caps should treat `mass_cap_multiplier=1.0` as diagnostic-only because it collapses interval shares to equal-time/uniform mass; production capped sweeps should use multipliers above 1.0.
- Matched no-rstar TVD variants must keep `signal_trace_key` in the selected-case payload; otherwise locked-test promotion silently resolves back to the canonical signal.
- LOB `_collect_calibration(...)` payloads can omit `info_growth_hardness_no_rstar_by_step` even when `residual_norm_by_step` and `disagreement_by_step` are present. Before calling `canonical_tvd_schedule_details(...)` for no-rstar TVD, synthesize the missing trace from those arrays and backfill its validation Spearman entry from calibration rows.
- Conditional-generation main-table runs should use `--backbone_manifest TVD-result/results/backbone_matrix/backbone_manifest.json` for the 20k matrix; the default shared-backbone root can be missing `sleep_edf`, and Windows here-strings can inject carriage returns into remote `OUT` paths.
- Figure collectors launched from `code/` must resolve relative backbone manifests against the project root; otherwise `sleep_edf` falls back to the incomplete shared-backbone root. Detached `ssh yzn3090` launches are safest with absolute Python, script, log, and output paths.
- For Isambard3 smoke submissions, prefer `sbatch <script>` with exported environment variables over PowerShell-piped batch heredocs or complex `--wrap` commands; CRLF and remote shell semicolon parsing can otherwise make Slurm reject the job or run setup on the login node.
- The PTG-vs-observed-gain diagnostic should not be promoted automatically: both info-growth PTG and local-defect PTG remained negatively correlated with observed forecast gains on the 20k matrix, so the script must gate paper-facing output on positive Spearman and otherwise emit `_diagnostic` figures.
- Detached `yzn3090` jobs launched through SSH should write their PID after confirming the Python child, not inside a complex `nohup ... & echo $!` chain; the chain can outlive the SSH timeout even when the job itself is running correctly.
- During the 2026-04-29 cleanup, active fixed-schedule evaluation moved to `diffusion_flow_time_reparameterization.py`; keep `TVD-result` only as stable storage naming and put canonical TVD methods, plans, and extras under the legacy manifest.
- Native hardness should use `info_growth_hardness_by_step`; avoid reintroducing `canonical_*`, `r_star`, or TVD-specific imports on the active path unless a legacy script is being inspected deliberately.
